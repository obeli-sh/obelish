# Round 3: Frontend Engineer — Final Consensus and Recommendations

## 1. Resolved Debates: Final Positions

### 1.1 Store Architecture: 2 Stores (ACCEPTED)

Tech lead and backend both argued for 2 stores instead of my original 4. PM advocated for 1. I accept the **2-store compromise** as the right call:

- **`workspaceStore`** — mirrors Rust's `WorkspaceState` as a read-only projection. Updated wholesale on each `workspace-changed` event. Contains workspaces, surfaces, panes, layout trees.
- **`uiStore`** — frontend-owned mutable state: focused pane ID, sidebar open/closed, notification panel open/closed, search state, scroll positions.

This works because:
1. Since Rust is the source of truth, workspace/layout/pane data always arrive together in a single event. There is no benefit to separating them into 3 stores — they are one unit.
2. Zustand selectors give us the fine-grained subscription I need to prevent re-renders. `useWorkspaceStore(s => s.workspaces[id].name)` only triggers when that specific name changes.
3. `uiStore` stays separate because it mutates at a different frequency (every keystroke for focus vs occasional structural changes) and never goes to Rust.

**No Zustand `persist` middleware.** Persistence is 100% Rust-owned. Frontend stores are ephemeral.

### 1.2 Coverage Threshold: 95% Gate (ACCEPTED WITH NUANCE)

Tech lead proposed 95% as the CI gate. QA wants 100%. PM wants 95% with 100% on business logic. I am aligning with the **tech lead's 95% gate** as the pragmatic choice, with the following frontend-specific targets:

| Code Category | Target | CI Gate | Rationale |
|--------------|--------|---------|-----------|
| Zustand stores | 100% | 100% | Pure logic, easy to test, bug-prone |
| Custom hooks (`useTerminal`, `useTauriEvent`, etc.) | 100% | 100% | Lifecycle bugs (memory leaks, stale subscriptions) live here |
| Components (behavioral tests) | 95%+ | 90% | Some JSX branches are genuinely low-value to cover |
| `tauri-bridge.ts` | 100% | 100% | The Rust-React contract — every function must be tested |
| CSS/theme files | N/A | N/A | Visual regression via Playwright, not line coverage |

The 90% component gate prevents CI from failing over an untested conditional CSS class, while the 100% store/hook/bridge gates protect the code where bugs actually matter.

QA: I hear your position on 100% everywhere. Here is my concession: **if a component's coverage drops below 95%, the PR author must justify it in the PR description.** This prevents lazy coverage drops while avoiding contrived tests for the last 5%.

### 1.3 E2E Retry Policy: 1 Retry for E2E Only (ACCEPTED)

QA insists on 0 retries. I pushed for 1 retry on E2E. The tech lead's Round 2 is silent on this, but the backend supports a pragmatic approach.

**Final position:**
- Unit and integration tests: **0 retries.** A flaky unit test is a broken test.
- E2E tests (Playwright + tauri-driver): **1 retry.** With mandatory investigation if a test retries more than 5% of runs. Any test exceeding this threshold is quarantined within 48 hours per QA's protocol.

This is a reasonable compromise. Playwright + WebDriver over a real Tauri app has inherent timing variability that unit tests do not. One retry catches transient failures without masking real bugs.

### 1.4 `workspace-changed` Event: Full State Replacement (ACCEPTED)

Backend's Round 2 provides a `WorkspaceChangedEvent` with `change_type` discriminator AND full `WorkspaceInfo` state. Tech lead explicitly endorsed this in Round 2 section 5. This is exactly what I need on the frontend.

**Final frontend handling:**
```typescript
// In workspaceStore initialization
listen<WorkspaceChangedEvent>('workspace-changed', (event) => {
  const { workspaceId, state } = event.payload;
  useWorkspaceStore.setState(prev => ({
    workspaces: { ...prev.workspaces, [workspaceId]: state }
  }));
});
```

The `changeType` field is useful for logging and analytics but the frontend does NOT branch on it — it always replaces the full workspace state. This keeps the handler trivially simple and testable.

**Tech lead's concern about 20+ pane serialization overhead:** The workspace state is a tree of IDs, strings, and a layout enum. Even with 20 panes, this is a few KB of JSON. Serialization is sub-millisecond. Not a concern.

### 1.5 Shared Protocol Crate: Defer to Phase 7 (ACCEPTED)

PM and I aligned on YAGNI. Backend wants it now. Tech lead wants it now.

**Compromise accepted from tech lead's Round 2:** The crate exists from Phase 0 as part of the Cargo workspace scaffold, but it starts nearly empty — just the shared error types and a few core IDs. It grows organically as types are needed by both `src-tauri` and `cli`. This avoids premature design while setting up the infrastructure path.

Practically, this means I consume types from `src/lib/generated/types.ts` (auto-generated by `ts-rs` from whichever crate the types live in). The frontend doesn't care whether types come from `obelisk-protocol` or `src-tauri` — `ts-rs` handles the generation either way.

### 1.6 Accessibility: Semantic HTML from Day 1, Formal a11y Post-MVP (ACCEPTED)

PM said defer to Phase 8. I originally wanted it from Phase 1. The compromise:

**What we do from Phase 1:**
- Semantic HTML elements (`<nav>`, `<main>`, `<button>`, `<input>`)
- `role="tablist"` / `role="tab"` for surface tabs (it is trivial to add and prevents incorrect screen reader behavior)
- Standard keyboard navigation (Tab/Shift+Tab focus flow)
- Visible focus indicators on all interactive elements
- `aria-label` on icon-only buttons

**What we defer to post-MVP:**
- Full WCAG 2.1 AA audit
- Screen reader testing with NVDA/VoiceOver
- xterm.js accessibility addon integration
- Automated a11y testing (axe-core in Playwright)

This is semantic HTML as a default practice, not a formal accessibility initiative. It costs almost nothing to do right from the start and prevents technical debt.

## 4. Performance Optimization Plan

### 4.1 PTY Data: Zero React Renders

The most critical performance path. PTY data NEVER enters React state:

```typescript
function useTerminal(paneId: string, ptyId: string) {
  const terminalRef = useRef<Terminal | null>(null);

  // Direct xterm.js write — no setState, no re-render
  useTauriEvent<PtyDataEvent>(`pty-data-${ptyId}`, (payload) => {
    const decoded = atob(payload.data);
    terminalRef.current?.write(decoded);
  });

  // ... rest of hook
}
```

### 4.2 Layout Resize: Memoization Strategy

```typescript

  // ...
});

// PaneWrapper also memoized — only re-renders when its specific pane changes

  // ...
```

### 4.3 Zustand Selectors for Fine-Grained Updates

```typescript

// BAD: Re-renders on any store change

```

### 4.4 WebGL Context Pooling

Phase 1 strategy for the WebGL context limit (8-16 per browser page):

1. **Visible panes** get the WebGL addon (GPU-accelerated rendering)
2. **Hidden panes** (inactive surfaces/tabs) get the Canvas addon (CPU rendering, no WebGL context)
3. When switching surfaces, swap addons: detach WebGL from hidden, attach to newly visible
4. If WebGL context creation fails (limit reached), fall back to Canvas silently

Implementation:
```typescript
function useTerminal(paneId: string, ptyId: string) {
  const [rendererType, setRendererType] = useState<'webgl' | 'canvas'>('webgl');

  // Try WebGL, fall back to Canvas
  useEffect(() => {
    try {
      const webgl = new WebglAddon();
      terminal.loadAddon(webgl);
    } catch {
      const canvas = new CanvasAddon();
      terminal.loadAddon(canvas);
      setRendererType('canvas');
    }
  }, []);
}
```

### 4.5 Terminal Detach/Re-Attach for Tab Switching

When a surface (tab) is hidden, its terminals must NOT be disposed:

```typescript
function useTerminal(paneId: string, ptyId: string) {
  const terminalInstance = useRef<Terminal | null>(null);

  const terminalRef = useCallback((el: HTMLDivElement | null) => {
    if (el && terminalInstance.current) {
      // Re-attach existing terminal to new DOM element
      terminalInstance.current.open(el);
      fitAddon.current?.fit();
    } else if (el && !terminalInstance.current) {
      // First mount — create terminal
      terminalInstance.current = new Terminal({ /* options */ });
      terminalInstance.current.open(el);
    }
    // When el is null (unmount), do NOT dispose — just let it detach
  }, []);

  // Dispose only on component tree removal (pane close), not tab switch
  useEffect(() => {
    return () => {
      // Only called when the component is truly removed from the tree
      terminalInstance.current?.dispose();
      terminalInstance.current = null;
    };
  }, []);
}
```

Key: React does NOT unmount components when switching surfaces — we use `display: none` on inactive surface content, keeping the React tree alive. Terminals detach from DOM but are not disposed.

## 7. Keyboard Shortcuts (Phase 2)

### 7.1 Shortcut Map

| Shortcut | macOS | Windows/Linux | Action |
|----------|-------|--------------|--------|
| Split horizontal | `Cmd+Shift+H` | `Ctrl+Shift+H` | `tauriBridge.pane.split(focusedPaneId, 'horizontal')` |
| Split vertical | `Cmd+Shift+V` | `Ctrl+Shift+V` | `tauriBridge.pane.split(focusedPaneId, 'vertical')` |
| Close pane | `Cmd+W` | `Ctrl+W` | `tauriBridge.pane.close(focusedPaneId)` |
| New workspace | `Cmd+N` | `Ctrl+N` | `tauriBridge.workspace.create()` |
| Switch workspace 1-9 | `Cmd+1-9` | `Ctrl+1-9` | `tauriBridge.workspace.focus(workspaces[n-1].id)` |
| Next surface | `Cmd+Tab` | `Ctrl+Tab` | `uiStore.nextSurface()` |
| Focus pane up | `Cmd+Up` | `Ctrl+Up` | `uiStore.focusAdjacentPane('up')` |
| Focus pane down | `Cmd+Down` | `Ctrl+Down` | `uiStore.focusAdjacentPane('down')` |
| Focus pane left | `Cmd+Left` | `Ctrl+Left` | `uiStore.focusAdjacentPane('left')` |
| Focus pane right | `Cmd+Right` | `Ctrl+Right` | `uiStore.focusAdjacentPane('right')` |
| Toggle notification panel | `Cmd+I` | `Ctrl+I` | `uiStore.toggleNotificationPanel()` |

### 7.2 Implementation

```typescript
// useKeyboardShortcuts registers at the window level
function useKeyboardShortcuts(shortcuts: ShortcutMap) {
  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      const mod = navigator.platform.includes('Mac') ? e.metaKey : e.ctrlKey;
      const key = normalizeKey(e);
      const match = shortcuts.find(s => s.mod === mod && s.shift === e.shiftKey && s.key === key);
      if (match) {
        e.preventDefault();
        e.stopPropagation();
        match.action();
      }
    };
    window.addEventListener('keydown', handler, { capture: true });
    return () => window.removeEventListener('keydown', handler, { capture: true });
  }, [shortcuts]);
}
```

Using `capture: true` so shortcuts fire before xterm.js captures the keystroke. The terminal should NOT receive shortcut keys.

### 7.3 Conflict Avoidance

Known conflicts to handle:
- `Ctrl+W` closes browser tabs — in Tauri, this is not a concern (no browser chrome), but `Cmd+W` on macOS closes the window by default. We need to prevent this via Tauri config or the `keydown` handler.
- `Ctrl+C` must NOT be captured — it is a terminal interrupt signal. Same for `Ctrl+D`, `Ctrl+Z`.
- `Ctrl+Tab` may conflict with Tauri's internal tab handling on some platforms — test early.

## 8. Frontend Error Handling Contract

Accepting backend's structured error proposal from Round 2:

```typescript
interface BackendError {
  kind: string;
  message: string;
}

// Error handling utility
function handleBackendError(error: unknown, context: string): void {
  const err = error as BackendError;
  switch (err.kind) {
    case 'PtyNotFound':
    case 'AlreadyKilled':
      // Stale reference — trigger workspace resync
      tauriBridge.workspace.list().then(syncAllWorkspaces);
      break;
    case 'SpawnFailed':
      // Show error in the pane's error boundary
      throw new Error(`Terminal failed to start: ${err.message}`);
    case 'PersistenceError':
      // Non-fatal — show toast
      showToast(`Save failed: ${err.message}`, 'warning');
      break;
    default:
      console.error(`[${context}] Backend error:`, err);
      showToast(err.message, 'error');
  }
}
```
